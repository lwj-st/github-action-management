# GitHub Action Management 产品需求文档 (PRD)

## 产品概述

一款面向开发者与运维团队的轻量级 GitHub Actions 本地管控桌面应用，提供直观的 UI 和强大的功能，简化 GitHub Actions 的日常管理和运维工作。

## 目标用户

- 开发者：需要频繁触发和监控 CI/CD 流程
- 运维团队：负责生产环境的部署和监控
- 技术团队：需要批量管理多个仓库的 Workflows

## 核心价值

1. **集中管理** - 在单个界面管理多个账户和仓库的 Workflows
2. **便捷触发** - 参数化触发，无需登录 GitHub 网页
3. **实时监控** - 实时查看运行状态和日志
4. **高效操作** - 一键取消、重跑，快速解决问题

## 功能需求

### 3.1 安全性要求

| 需求 | 描述 | 优先级 |
|------|------|--------|
| 本地存储 | Token 和配置仅存储在本地，不上传云端 | P0 |
| 密钥环集成 | 支持系统密钥环（Keychain/Credential Manager） | P1 |
| 权限最小化 | Token 仅申请必要权限（repo, workflow, actions） | P0 |
| Token 过期提醒 | 提前 7 天提醒用户刷新 Token | P1 |
| 操作审计 | 记录所有操作的时间戳和操作人 | P1 |
| 防滥用 | 遵循 API 速率限制，智能重试 | P0 |

### 3.2 实用性 & 性能

| 需求 | 描述 | 优先级 |
|------|------|--------|
| 智能缓存 | 缓存仓库和 Workflow 列表，减少 API 调用 | P1 |
| 内存占用 < 200MB | 常规操作下内存占用控制 | P1 |
| 弱网适配 | 在网络不稳定时提供友好提示和重试机制 | P2 |
| 快捷操作 | 支持快捷键（触发、取消、刷新） | P2 |
| 并发限制 | 同一时间最多触发 10 个 Workflow | P2 |

### 3.3 跨平台兼容性

| 需求 | 描述 | 优先级 |
|------|------|--------|
| macOS 支持 | 支持 macOS 10.15+ | P0 |
| Windows 支持 | 支持 Windows 10+ | P0 |
| Dark/Light 模式 | 支持暗色主题（默认） | P1 |
| 自动更新 | 自动检测并提示更新 | P2 |
| 多语言支持 | 支持英文界面 | P2 |

### 3.4 UI/UX 美观度

| 需求 | 描述 | 优先级 |
|------|------|--------|
| 现代设计系统 | 使用 Material Design 或类似设计语言 | P1 |
| 暗色主题 | 默认暗色主题，保护眼睛 | P0 |
| 数据可视化 | 使用图表展示运行历史趋势 | P2 |

## 非功能性需求

### 4.1 性能要求

- **启动时间** < 2 秒
- **API 响应** < 3 秒（正常网络）
- **日志加载** < 1 秒（单作业日志）
- **内存占用** < 200MB（常规操作）
- **CPU 占用** < 10%（空闲状态）

### 4.2 可靠性要求

- **可用性** ≥ 99%
- **崩溃率** < 0.1%
- **数据丢失** 0 次
- **自动恢复** 失败操作自动重试

### 4.3 可维护性要求

- **代码覆盖率** ≥ 80%
- **文档完整度** 100%
- **单元测试** 核心模块全覆盖
- **E2E 测试** 主要流程全覆盖

### 4.4 安全性要求

- **加密存储** Token 使用 AES-256 加密
- **传输安全** 仅使用 HTTPS
- **权限验证** 每次 API 调用验证权限
- **漏洞扫描** 定期扫描依赖漏洞

## 用户故事

### US-001: 添加 GitHub 账户

**作为** 开发者  
**我想要** 添加我的 GitHub 账户  
**以便于** 管理我的仓库和 Workflows  

**验收标准**:
- [ ] 用户可以输入账户名称和 PAT Token
- [ ] Token 本地加密存储
- [ ] 验证 Token 有效性
- [ ] 显示账户列表

### US-002: 触发 Workflow

**作为** 开发者  
**我想要** 触发一个 Workflow 并传入参数  
**以便于** 手动执行部署或测试  

**验收标准**:
- [ ] 自动识别 `workflow_dispatch` 输入
- [ ] 根据类型生成对应输入控件
- [ ] 必填参数验证
- [ ] 触发后显示运行 ID
- [ ] 自动跳转到运行列表

### US-003: 查看运行状态

**作为** 运维人员  
**我想要** 实时查看 Workflow 运行状态  
**以便于** 了解执行进度和结果  

**验收标准**:
- [ ] 轮询状态更新（3 秒间隔）
- [ ] 不同状态使用不同颜色标识
- [ ] 显示运行详情（分支、SHA、时间）
- [ ] 点击可查看日志

### US-004: 取消运行

**作为** 运维人员  
**我想要** 取消正在运行的 Workflow  
**以便于** 避免不必要的资源消耗  

**验收标准**:
- [ ] 提供取消按钮
- [ ] 取消请求 ≤ 2 秒生效
- [ ] 状态同步更新为 cancelled
- [ ] 显示取消确认提示

### US-005: 重新运行

**作为** 开发者  
**我想要** 重新运行失败的 Workflow  
**以便于** 快速验证修复  

**验收标准**:
- [ ] 提供重新运行按钮
- [ ] 提供重跑失败作业选项
- [ ] 保留原始运行配置
- [ ] 重新运行后状态实时更新

### US-006: 查看日志

**作为** 开发者  
**我想要** 查看 Workflow 运行日志  
**以便于** 排查问题  

**验收标准**:
- [ ] 显示完整日志内容
- [ ] 支持关键词过滤
- [ ] 支持日志复制
- [ ] 支持日志下载

## API 集成要求

### 5.1 GitHub REST API

| 接口 | 用途 | 频率限制 |
|------|------|----------|
| GET /user | 获取当前用户 | 5000/h |
| GET /user/repos | 获取用户仓库 | 5000/h |
| GET /repos/{owner}/{repo}/actions/workflows | 获取 Workflows | 5000/h |
| POST /repos/{owner}/{repo}/actions/workflows/{workflow_id}/dispatches | 触发 Workflow | 5000/h |
| GET /repos/{owner}/{repo}/actions/runs | 获取运行列表 | 5000/h |
| GET /repos/{owner}/{repo}/actions/runs/{run_id} | 获取运行详情 | 5000/h |
| POST /repos/{owner}/{repo}/actions/runs/{run_id}/cancel | 取消运行 | 5000/h |
| POST /repos/{owner}/{repo}/actions/runs/{run_id}/rerun | 重新运行 | 5000/h |
| POST /repos/{owner}/{repo}/actions/runs/{run_id}/rerun-failed-jobs | 重跑失败作业 | 5000/h |
| GET /repos/{owner}/{repo}/actions/jobs/{job_id}/logs | 获取日志 | 5000/h |
| GET /repos/{owner}/{repo}/actions/runs/{run_id}/artifacts | 获取制品 | 5000/h |

### 5.2 错误处理

| 错误码 | 含义 | 处理方式 |
|--------|------|----------|
| 401 | 认证失败 | 提示 Token 无效或过期 |
| 403 | 权限不足 | 提示 Token 权限不足 |
| 404 | 资源不存在 | 提示资源不存在 |
| 422 | 请求参数错误 | 提示具体错误信息 |
| 429 | 请求频率过高 | 自动重试，显示等待时间 |

- **Job**: Workflow 中的一个任务单元
- **Artifact**: Workflow 产生的制品文件
